# Modern Python & Zero-Fat Coding Rules

## 概要

本報告書では、Astral 製ツールチェーン（uv, Ruff, ty）および Pydantic v2 を中核とした最新の開発フローと、エラーハンドリングやドキュメンテーションに対する過剰な実装を排除する「Zero-Fat」原則について、その背景、実装メカニズム、および組織的な利点について詳述する。

---

## 1. ツールチェーンの近代化と再現性の確保

現代の Python 開発において、開発環境の構築と依存関係の管理はプロジェクトの成否を分ける基盤である。

### 1.1 パッケージ・プロジェクト管理: uv

uv は Rust で記述された超高速なパッケージマネージャーであり、従来の pip, poetry, pyenv 等の機能を統合する。

- **決定論的な再現性**: `uv.lock` による厳密な状態管理。 CI/CD 環境における「環境の揺らぎ」を完全に排除する。
- **標準操作**: `uv add`, `uv sync`, `uv run` を基本とする。
- **Python インタープリタ管理**: `.python-version` に基づき自動インストール・利用を行う。

| 管理項目 | 従来のアプローチ | Modern (uv) によるアプローチ |
| :--- | :--- | :--- |
| 解決速度 | 遅い（バックトラック発生） | 極めて高速（並列処理） |
| ロックファイル | requirements.txt / poetry.lock | uv.lock (ユニバーサル) |
| 環境構築 | venv + pip install | uv sync (宣言的同期) |
| バージョン管理 | pyenv 等 | uv python (ネイティブ管理) |

### 1.2 静的解析とフォーマット: Ruff

Ruff は、コードの品質を均一に保つための Linter & Formatter である。

- **統合**: 900 以上のルールを単一バイナリでサポートし、`pyproject.toml` で一元管理される。
- **自動修正**: `uv run ruff check --fix` により、規約違反の多くを機械的に排除する。
- **フォーマット**: `uv run ruff format` で Black 互換のスタイルを強制する。

---

## 2. Pydantic v2 によるデータモデル

データ整合性の保証はシステムの堅牢性を維持するための核心的課題である。

- **厳格な型定義**: `BaseModel` と `Annotated` を活用し、バリデーションロジックをモデルに内包させる。
- **不変性の優先**: `ConfigDict(frozen=True)` を設定し、予期せぬ副作用を排除する。
- **パフォーマンス**: Rust コアによる高速化。 `model_validate_json` により中間オブジェクト生成を回避しメモリ効率を高める。

## 3. 型チェック: Pylance と ty

### 3.1 typing.Any の完全禁止

Any は型システムの健全性を損なうため、例外なく禁止する（Ruff ANN401）。どうしても型が確定しない場合は `object` 型を使用するか、ジェネリクスを用いるべきである。

### 3.2 高速型チェッカー ty (Next Phase)

Astral 製の `ty` は、従来のチェッカーより 10〜80 倍高速なフィードバックを可能にする次世代ツールである。本プロジェクトでも試験的に導入を検討する。

---

## 4. Zero-Fat 原則: 本質的なロジックへの回帰

冗長なコード、過剰なエラーハンドリング、維持されないドキュメント（脂肪）を徹底的に排除する。

### 4.1 Fail Fast (例外処理の禁止)

`try-except` によるエラー隠蔽を禁止する。不具合は即座にクラッシュさせ、根本原因を早期に特定・修正する。インフラ側での自己修復やリトライに委譲し、アプリケーション内部での複雑なリカバリーロジックを排除する。

### 4.2 Self-Documenting (コメント・docstring の排除)

- **命名による意図伝達**: 名前だけで役割と副作用を完全に説明する。
- **型システムによる仕様定義**: 厳格な型定義を関数の「契約」として機能させる。
- **論理構造**: 関数のサイズを最小化し、コードの流れ自体をドキュメントとする。

### 4.3 Success Path Only (正常系のみの記述)

リトライ、冗長なタイムアウト、例外的な条件分岐をメインフローから排除する。関数本体は「すべてが正常に進んだ場合」のパスに集中させる。

---

## 5. 実装パターン

### 5.1 デコレータによる関心の分離
ロギング、認証、レート制限等の「横断的な関心事」はデコレータに分離し、関数本体はビジネスロジックに集中させる。

### 5.2 Zero-Hardcoding (外部構成)
すべての設定値、閾値、API エンドポイント等は `config.yaml` 等の外部ファイルに集約し、コード内への直接記述（ハードコード）を排除する。

---

## 6. 経済性とインパクト

Zero-Fat 原則の採用は、単なる技術的選択ではなく組織の生産性に寄与する。人的労力を削減するためにクラウドの計算資源を活用し、開発スピードを上げ、意思決定のループを加速させることが真の ROI 向上に繋がる。
